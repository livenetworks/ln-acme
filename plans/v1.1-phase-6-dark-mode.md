# Plan: v1.1 Phase 6 — Dark Mode

**Source roadmap:** `plans/design-system-v1.1-roadmap.md` § Phase 6
**Branch:** `v1.1/phase-6-dark-mode`
**Depends on:** Phase 2 (neutral scale), Phase 4 (cool shadows). Phase 3
(typography) nice-to-have.
**Unblocks:** Phase 7 components (they're verified under both themes)

---

## Context

ln-acme v1.0 has `scss/config/_theme.scss` as a 13-line empty
placeholder. No dark mode exists. For a B2B library in 2026 this is
an expected baseline.

Phase 2 built the neutral scale. Phase 4 built cool-tinted shadows.
Dark mode is now an **inversion** of the neutral scale and a
**re-tuning** of shadows and primary tint layers. Most of the work is
validating that each component reads well under both themes — the
infrastructure is already there.

## Pre-resolved ambiguities

### A. Activation strategy

Dual: explicit attribute + system preference.

```scss
// Explicit, always wins
[data-theme="dark"] { /* dark overrides */ }

// System preference, respected unless light is explicitly forced
@media (prefers-color-scheme: dark) {
	:root:not([data-theme="light"]) { /* dark overrides */ }
}
```

Rationale: users who force light mode on a dark-system device get
their override respected. Users who set `data-theme="dark"` on a
light-system device get their override respected. The default
behavior on no-opt-in is: follow the system.

**Open question #2 in the roadmap is decided here.** If the user
rejects auto-apply, revert to attribute-only by dropping the
`@media` block.

### B. Dark neutral scale

Inverted from Phase 2. The dark-900 is the lightest (used for
primary text) and the dark-50 is the darkest (the body surface).

```
Light mode → Dark mode

50:  220 20% 98%  →  220 20%  8%
100: 220 16% 96%  →  220 16% 12%
150: 220 14% 93%  →  220 14% 18%
200: 220 13% 91%  →  220 13% 24%
300: 220 13% 83%  →  220 13% 36%
400: 218 11% 65%  →  218 11% 52%    (keeps mid-tone close)
500: 220  9% 46%  →  220  9% 60%
600: 220 11% 34%  →  220 11% 72%
700: 220 14% 24%  →  220 14% 82%
800: 220 20% 15%  →  220 20% 90%
900: 221 39% 11%  →  221 30% 97%    (slightly desaturated — pure 39% is harsh on dark)
```

Semantic tokens — NONE remap. Because Phase 2 made every semantic
token reference `var(--color-neutral-*)`, flipping the scale flips
every downstream token automatically. This is the payoff of Phase 2.

### C. Shadow re-tuning for dark

On dark surfaces, the Phase 4 cool-tinted shadows become nearly
invisible (light shadow on dark background = no contrast). Re-tune to
solid black with a boosted alpha:

```scss
--shadow-xs:
	0 1px 2px 0 hsl(0 0% 0% / 0.32);

--shadow-sm:
	0 1px 3px 0 hsl(0 0% 0% / 0.40),
	0 1px 2px -1px hsl(0 0% 0% / 0.28);

--shadow-md:
	0 4px 12px -2px hsl(0 0% 0% / 0.48),
	0 2px 4px -2px hsl(0 0% 0% / 0.32);

--shadow-lg:
	0 12px 24px -6px hsl(0 0% 0% / 0.56),
	0 8px 12px -4px hsl(0 0% 0% / 0.40);

--shadow-xl:
	0 24px 48px -12px hsl(0 0% 0% / 0.64),
	0 12px 24px -6px hsl(0 0% 0% / 0.48);

--shadow-2xl:
	0 36px 72px -18px hsl(0 0% 0% / 0.72),
	0 18px 36px -9px hsl(0 0% 0% / 0.56);

--shadow-inner:
	inset 0 2px 4px 0 hsl(0 0% 0% / 0.32);
```

### D. Primary tint layers

`--color-primary-light: 232 75% 93%` and `--color-primary-lighter:
232 75% 97%` are near-white — unreadable on dark backgrounds. Re-tune:

```scss
--color-primary-light:   232 60% 22%;
--color-primary-lighter: 232 50% 15%;
```

Same for `--color-bg-error: 0 86% 97%` → `--color-bg-error: 0 40% 15%`.

### E. Accessibility targets

- Body text (`neutral-900` → `neutral-50` dark background): must be
  ≥7:1 (AAA).
- Secondary text (`neutral-500` → `neutral-50`): must be ≥4.5:1 (AA).
- Focus ring: ≥3:1 non-text contrast (WCAG 1.4.11).
- Primary button bg on body surface: ≥3:1 contrast.

Verify using a contrast checker against the computed HSL values.

### F. Opt-out / demo toggle

`demo/admin/index.html` gets a simple consumer-side JS snippet to
toggle `[data-theme]` on the root. Library does NOT ship a toggle
component — consumers write their own.

### Step 3 — Consumer theme toggle in demo (Commit 2)

Edit `demo/admin/index.html`. In the `<head>` or before `</body>`,
add a self-contained script block (consumer-side example — NOT a
library component):

```html
<script>
	// Consumer-side theme toggle example (library does not ship this)
	(function () {
		const KEY = 'ln-demo-theme';
		const root = document.documentElement;

		const stored = localStorage.getItem(KEY);
		if (stored === 'dark' || stored === 'light') {
			root.setAttribute('data-theme', stored);
		}

		const toggle = document.querySelector('[data-demo-theme-toggle]');
		if (toggle) {
			toggle.addEventListener('click', () => {
				const current =
					root.getAttribute('data-theme') ||
					(matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light');
				const next = current === 'dark' ? 'light' : 'dark';
				root.setAttribute('data-theme', next);
				localStorage.setItem(KEY, next);
			});
		}
	})();
</script>
```

Add a trigger button in the page header:

```html
<button type="button" data-demo-theme-toggle aria-label="Toggle dark mode">
	<svg class="ln-icon" aria-hidden="true"><use href="#ln-moon"></use></svg>
</button>
```

This is a demo-only example. Its purpose: let the user verify the
theming behavior visually.

### Step 5 — Document theming (Commit 3)

Create `docs/css/theming.md`:

```markdown
# Theming

File: `scss/config/_theme.scss`. Added in v1.1.

## Philosophy

ln-acme v1.1 supports a single dark-mode variant plus any number of
consumer-defined themes. Dark mode is built by **inverting the
neutral scale** — every semantic token references
`var(--color-neutral-*)`, so flipping the scale flips every downstream
token automatically.

Consumer projects can re-theme at any scope:

```scss
// Tenant-specific primary color
.tenant-acme { --color-primary: 340 75% 52%; }

// Section-specific neutral override
.print-preview { --color-bg-body: var(--color-neutral-100); }
```

## Dark mode activation

Three activation paths; any of them enables dark:

### 1. Explicit attribute (always wins)

```html
<html data-theme="dark">
```

User preference stored client-side (`localStorage`, cookie, etc.).
Library does not persist preference; consumer does.

### 2. System preference (auto)

```html
<html>
```

When the OS is in dark mode and no `data-theme` attribute is set,
ln-acme automatically applies dark tokens via
`@media (prefers-color-scheme: dark)`.

### 3. Opt-out of auto

```html
<html data-theme="light">
```

Forces light mode even on a dark-system device. Use when the user
has explicitly chosen light.

## Consumer toggle example

See `demo/admin/index.html` for a minimal vanilla JS toggle. Key
points:

```javascript
// Read current effective theme
const current = document.documentElement.getAttribute('data-theme') ||
	(matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light');

// Force a specific theme
document.documentElement.setAttribute('data-theme', 'dark');

// Clear override (return to system preference)
document.documentElement.removeAttribute('data-theme');
```

## What dark mode overrides

- **Neutral scale:** all 11 steps (`50` → `900`), inverted.
- **Primary tint layers:** `--color-primary-light`, `--color-primary-lighter`.
- **Error background:** `--color-bg-error`.
- **Shadows:** solid black, higher alpha.

All semantic tokens (`--color-text-primary`, `--color-bg-body`,
`--color-border`, etc.) reference the neutral scale and flip
automatically.

## What dark mode does NOT override

- **Primary hue** — stays the same brand color.
- **Status hues** (success, error, warning, info) — stay the same.
- **Typography** — same scale.
- **Spacing / layout** — identical.

## Per-component dark tuning

**You should not need it.** If a component looks wrong in dark mode,
the bug is almost always a hardcoded color somewhere in the
component — it should reference a token instead. Only fall back to
per-component dark overrides for genuinely unique cases (e.g., a
hero illustration with baked-in shadows).

## Extending

Custom theme:

```scss
[data-theme="high-contrast"] {
	--color-neutral-900: 0 0% 100%;
	--color-neutral-50:  0 0% 0%;
	--color-border:      var(--color-neutral-900);
}
```

Or tenant branding:

```scss
.tenant-acme {
	--color-primary: 340 75% 52%;
	--color-primary-hover: 340 75% 42%;
}
```

## Accessibility

- Body text ≥ 7:1 (AAA)
- Secondary text ≥ 4.5:1 (AA)
- Focus ring ≥ 3:1 non-text (WCAG 1.4.11)

The default neutral-900/50 flip clears all three. Custom themes must
re-verify with a contrast checker.
```
